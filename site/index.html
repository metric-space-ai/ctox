What CTOX is

A CLI daemon around agent turns.

CTOX is not primarily a desktop app. It is a command-line runtime: start the service with ctox start, send work with ctox chat, and let the service decide whether to run, queue, review, block, or continue that work later.

Service first The CLI talks to a persistent local service instead of treating every prompt as a disposable process.
Durable state Queue items, plans, schedules, governance events, secrets, continuity, and mission state live in runtime/ctox.sqlite3.
Execution slices Work is leased, executed through the embedded Codex harness, reviewed, persisted, and followed up when it is not actually done.

How it feels in use

You operate CTOX from the terminal, but it behaves like a running service.

Submit

ctox chat, queue items, plans, schedules, tickets, and communication channels all feed the same service loop.

Coordinate

CTOX tracks what is pending, leased, blocked, failed, handled, or waiting for owner approval.

Resume

Context health, compaction, continuity refreshes, and mission sync keep long work from depending on one perfect chat transcript.

ctox start
ctox chat "ship the release" --wait
ctox queue list --status pending
ctox plan tick
ctox runtime switch <model> quality

Compared with agent CLIs

CTOX is the runtime around the worker.

Codex-style agents

They are strongest at an interactive turn: inspect a repo, edit files, run commands, explain the result. CTOX keeps that kind of turn, but wraps it in a service, queue, database, watchdogs, and recovery rules.

Claude Code, OpenCode, OpenClaw, Hermes-style tools

The difference is not another chat box. CTOX is built for lifecycle: intake, lease, execute, review, persist evidence, schedule follow-up, and continue when the first slice is not enough.

CTOX CLI

The CLI is the operator interface for that runtime: start or stop the daemon, switch models, inspect status, repair queues, manage plans, store secrets, and keep autonomous work bounded by explicit governance.
