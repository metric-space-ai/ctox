An agentic daemon for Dev, DevOps, and Ops work.

You have servers, services, releases, incidents, migrations, and maintenance tasks. CTOX keeps that work organized, uses tickets and knowledge to build context, then dispatches agents to execute the next slice.

What CTOX is

The daemon around technical operations.

CTOX is for ongoing work on real systems. It owns the operating loop around tickets, plans, schedules, knowledge, communication, review, and follow-up. Coding agents are one execution layer inside that loop.

Work intake Important Dev, DevOps, and Ops work can enter through tickets, queues, schedules, plans, or communication channels.
Knowledge builder CTOX preserves findings, decisions, blockers, evidence, and context so the next run does not start cold.
Agent execution When work is runnable, CTOX sends a focused slice through the coding-agent stack and local tools.
Follow-up Incomplete, blocked, failed, or review-rejected work stays visible until it is handled or handed back.

How it works

CTOX runs the operating loop around your systems.

1. Track

CTOX watches the work backlog: tickets, plans, queues, schedules, channel input, and mission state.

2. Build context

It pulls together prior decisions, known blockers, relevant knowledge, current state, and expected next action.

3. Execute

It dispatches the next runnable slice to agents and tools: code changes, checks, scripts, browser work, or documentation.

4. Close or continue

It records the result, updates tickets or knowledge, and keeps a follow-up when the work is not done.

Compared with coding agents

The coding agent is the worker. CTOX owns the ops loop.

OpenAI Codex, Claude Code, OpenCode

These tools are strongest while a person is actively steering code work. CTOX is built for important work that must stay organized across many runs.

OpenClaw, Hermes-style agent runners

CTOX focuses on the daemon layer: keep the goal, ticket, knowledge, current state, next slice, result, and handoff.
