# Deterministic Onboarding Plan

CTOX must not improvise onboarding per source system.

Unless an operator explicitly asks for a plan change, CTOX should work through these stages in order and only stop when a real blocker or a missing primitive prevents the next stage.

## Stage 1: Attach and Validate

Required actions:
- test adapter access
- inspect adapter capabilities
- sync the source
- create or refresh the visible onboarding guide

Done when:
- the adapter responds
- the source is mirrored
- the onboarding guide exists

## Stage 2: Observe and Baseline

Required actions:
- inspect source knowledge
- inspect open CTOX self-work
- confirm the source is still in observe-only behavior until a real active source skill exists

Done when:
- CTOX has a current source profile
- no unjustified onboarding spam has been created

## Stage 3: Build the First Desk Skill

Required actions:
- export canonical ticket history
- build and evaluate a desk-specific source skill
- activate the skill only if evaluation gates pass
- refresh the onboarding guide

Done when:
- an active source skill is bound

## Stage 4: Open the First Execution Gap

Required actions:
- check whether confirmed knowledge-validation events already exist
- if not, create exactly one justified self-work item for the first execution supplement or runbook enrichment
- assign it to CTOX when supported
- note the next concrete source or system evidence needed
- refresh the onboarding guide

Done when:
- at least one explicit execution-enrichment work item exists or the first confirmed runbook already exists

## Stage 5: Controlled Real Work

Required actions:
- use the bound source skill on real ticket work
- keep execution gated through dry-run, approval, execution, verification, and writeback
- publish only real knowledge events such as runbook confirmation or correction
- refresh the onboarding guide

Done when:
- at least one confirmed runbook application exists

## Stage 6: Release Toward Bounded Autonomy

Required actions:
- keep repeating confirmed runbook applications
- correct or split weak runbooks
- close the onboarding guide only once the normal ticket path is now primarily driven by the active source skill

Done when:
- multiple confirmed runbook applications exist
- onboarding is no longer the main operating skill
