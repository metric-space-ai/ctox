# Project Description Reference Archetype

This archetype is derived from the project-description references in
`/Users/michaelwelsch/Downloads/OneDrive_1_9/`. It is mandatory for
`report_type_id=project_description`.

The references are not research reports. They are applicant-side funding
documents. Public research, company pages, reference material and review
comments are working material only; the final Word file must read like a
Vorhabensbeschreibung written for a funding/financing application.

## Reference corpus pattern

The observed corpus has seven examples:

- Walter Gott / Nutzeis production site: company group, legal tables, history,
  product/customer segments, innovation project, current bottleneck,
  development goals, project cost and implementation period.
- Lubeca / Marzipan production: legal table, company/product history,
  quality and product positioning, project introduction, current
  production/site bottleneck, development goal, measures, costs and period.
- Nordhessen Arena / event and park projects: company/property context,
  venue story, status quo, digital/service innovation, implementation
  requirements and project frame.
- A.-W. HEIL & SOHN / CRM and PIM: company profile, product portfolio,
  customer/service organisation, two innovation projects, each with current
  state/problem, development goal, implementation logic, costs and period.
- mind2move / B2B used-car platform: company founding, ownership,
  group structure, market performance, platform logic, project introduction,
  status quo/problem, AI/digital target model, measures, cost and period.
- Peters Stahlbau / laser process: legal profile, group structure, history,
  production capabilities, project introduction, current process problem,
  development goal, implementation challenges, cost and period.

These references share a stable process sequence, even when headings differ.

## Required document sequence

Use this sequence unless the operator explicitly demands a different form:

1. Title / programme context, if known.
2. Company/legal profile:
   applicant name, legal form, seat/address, register, capital, management,
   ownership/shareholder context where available.
3. Company profile:
   what the company does, for whom, with which products/services, at which
   sites and with what relevant capabilities.
4. Company development / history:
   foundation, growth path, milestones, group structure, prior innovations,
   capacity or market development.
5. Products, services and customer segments:
   named product lines, service model, target groups, reference segments,
   operating model and value chain.
6. Project introduction:
   one clear project sentence beginning from the company situation, not from
   generic innovation language.
7. Current state / problem area:
   current process/system/site/product baseline, concrete bottlenecks,
   operational consequences and why the existing setup is no longer enough.
8. Development goal:
   the target state after implementation, phrased as a future operating
   model with capabilities, not as a vague vision.
9. Innovation and market/state-of-the-art delimitation:
   why this is more than standard procurement, replacement or routine
   capacity expansion.
10. Challenges and planned measures:
    implementation dependencies, work packages, involved functions,
    technical/organisational measures, risks and milestones.
11. Project costs and implementation period:
    project cost, eligible cost logic if available, budget blocks, duration
    and start/end or project-release logic.
12. Economic benefit and use:
    economic mechanisms: capacity, margin, service quality, scalability,
    resilience, customer retention, sustainability, Standort- and
    competitiveness effects.

If the operator supplies an eight-chapter list, keep that user structure only
as a packaging constraint. The content above still has to appear. For example,
company/legal profile, history, products and customer segments all belong
inside "Unternehmensausgangslage"; "Projektumfang" still has to contain
"Projektkosten" and "Umsetzungszeitraum" in reference style.

## Heading vocabulary

The references strongly prefer these client-facing terms:

- `Gesellschaftsrechtliche Verhältnisse`
- `Unternehmensprofil`
- `Historie` or `Unternehmensentwicklung`
- `Produkte`, `Produktvielfalt`, `Kundensegmente`, `Veranstaltungskonzept`
- `Vorstellung des Innovationsprojektes` or `Vorstellung des Digitalisierungsprojektes`
- `Allgemeine Einleitung`
- `Derzeitiger Stand / Problembereich` or `Aktueller Stand und Problemstellung`
- `Entwicklungsziel`
- `Herausforderungen bei der Implementierung`
- `Projektkosten`
- `Umsetzungszeitraum`

Do not make the final document look like a study by using section names such
as "Quellenlage", "Recherchebasis", "Evidenz", "Methodik", "Screening",
"Bewertung nach Quellen" or "Literatur".

## Required tables

Use native Word tables sparingly, but at least one table should appear when
the facts are known:

- Company/legal table: applicant, address/seat, register, capital,
  management and ownership/shareholders.
- Project frame table: duration/period, status, budget/project costs,
  cost blocks.

The visible table must not contain a research column such as `Quelle`,
`Herkunft`, `Evidence`, `run_id` or URL. Source provenance stays in the
internal evidence register.

## Writing style

Write as the applicant or close third person:

- "Die Gesellschaft ..." / "Das Unternehmen ..." / "Wir ..." are all
  acceptable when consistent.
- Paragraphs are explanatory and concrete, not academic.
- Publicly researched facts are absorbed into the text without citation.
- Do not evaluate the company from outside. Present the project logic.
- Avoid generic funding language unless it is anchored in a company-specific
  bottleneck or measure.

The text must make sense if printed as an application attachment with no
visible bibliography.

## Red flags

A project-description draft fails this archetype when:

- it starts directly with the innovation without first building the company
  and capability context;
- it has no legal/company profile even though such facts are available;
- it has no history, product/service and customer/segment material;
- it describes market research or literature instead of a planned project;
- it uses source IDs, links, DOI strings, evidence wording or QA/tool terms;
- it says "Projektumfang" but does not clearly state project costs and
  implementation period;
- it contains generic AI/digitalisation prose that could fit any applicant;
- it lists facts but does not connect them to the project problem and
  development goal.
