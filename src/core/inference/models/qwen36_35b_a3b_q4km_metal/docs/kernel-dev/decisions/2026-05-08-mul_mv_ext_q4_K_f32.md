# Decision Record — mul_mv_ext_q4_K_f32 row-batched matvec (stage 3.4)

## Decision

```text
date:           2026-05-08
experiment:     wire vendored kernel_mul_mv_ext_q4_K_f32_r1_{2,3,4,5}
                from upstream ggml-metal; autotune r1ptg + nxpsg + nsg
                across the N=8..32 gap left between mul_mv (N=1 best)
                and mul_mm (N≥32 best).
decision:       opt-in    for narrow shapes (M ≤ 512) at N ∈ [8, 31]
                rejected  for wide shapes (M ≥ 2048) — mul_mm always
                          wins for them at N ≥ 8
accepted_env:   QWEN36_KERNEL_SELECTOR_NARROW_M=mul_mv_ext_r1_4_nxpsg_8
                QWEN36_KERNEL_SELECTOR_NARROW_M_MIN_N=8
                QWEN36_KERNEL_SELECTOR_NARROW_M_MAX_N=24
rejected_env:   no global default — only narrow-M shapes opt in
```

## Summary

The ext-row-batched matvec family fills the small-N gap, but only
for shapes where mul_mm's tile size (NRA=64 × NRB=128 × NK=32) is too
big to amortize. For Qwen3.6's wide projections (Q-proj 4096×2048,
O-proj 2048×4096) mul_mm wins at every N≥8; for narrow projections
(KV-proj 512×2048, FFN 512×2048) mul_mv_ext_r1_4 with nxpsg=8 wins by
~10–20 % until N=32 where they converge.

## Correctness

```text
Q-proj 4096×2048×r1 (r1 ∈ {2,3,4,5}, nxpsg ∈ {2,4,8}):
  max_abs ≤ 1e-3 across all 12 (r1, nxpsg) combinations.
```

This kernel runs everything in float lanes (no f16 dequant intermediate),
so its precision matches the matvec verifier's 2-ULP envelope, not the
matmat's 0.05 envelope.

## Selector logic (accepted)

```rust
fn pick_q4_k_f32_kernel(m: usize, n: usize) -> Q4KKernelChoice {
    use Q4KKernelChoice::*;
    match (m, n) {
        // Decode hot path
        (_, 1) => MulMv { nsg: 4 },
        // Narrow projections at small batch — ext wins by ~15 %
        (m, n) if m <= 512 && n >= 2 && n <= 24 => {
            // r1=4 wins as long as n ≥ r1; at n=2 use r1=2 etc.
            let r1 = (n as u32).min(4);
            MulMvExt { r1ptg: r1, nsg: 4, nxpsg: 8 }
        }
        // Otherwise mul_mm wins (wide M shapes from N=8, narrow from N=24+)
        _ => MulMm,
    }
}
```

This selector is recorded as the **accepted decode/prefill kernel
mix** for the dense projections — the MoE expert path (stage 3.6)
uses `kernel_mul_mv_id_q4_K_f32` instead and bypasses this selector.

## Why this decision is honest

- mul_mm with HAS_TENSOR / metal4.0 is more aggressive than I expected:
  it wins on wide shapes from N=8 onwards, not just from N=32.
  Stage 3.3's "N=32" threshold was conservative; updated here.
- ext kernel's narrow-M win is real (~15 %) but bounded — only matters
  for KV/FFN projections in the N=8..24 batched-decode regime that
  speculative decoding or beam search would hit. Pure-greedy decode
  (N=1) and prefill (N≥128) skip this regime entirely.
- The `_R5` variant never won — for our shapes r1=4 with high nxpsg
  saturates the matvec hot loop.
