## Why this matters for Qwen3.6 decode

Qwen3.6-35B-A3B has **256 experts top-8 per token** plus an always-on
shared expert. A naïve per-expert dispatch loop launches **256**
mul_mv kernels per layer per token where only **8** carry weight; a
naïve "select then run" loop launches **8** mul_mv kernels per layer
per token. The vendored `kernel_mul_mv_id_q4_K_f32` collapses those
8 launches into **one** because the kernel itself reads the chosen
expert IDs from a buffer and walks 8 expert weight matrices in one
pass.

For Qwen3.6 the per-token MoE FFN cost is therefore:
- 1 Rust router call (softmax + top-k over 256 logits, < 1 µs)
- 3 indexed-matvec dispatches (gate + up + down) — one per FFN matmul
- 1 in-Rust weighted-sum across slots
- Per-layer total kernel launches: 3 (vs the naïve 24)

## Correctness evidence

```text
shape: n_experts=16  top_k=4  m=128  k=256  n_tokens=1
GPU vs CPU max_abs              = 7.629e-6   (~7 ULP for f32)
weighted-sum end-to-end max_abs = 3.815e-6   (3 ULP)
```

The CPU reference is dequantize-q4_K → f32 → naive matmul → weighted
sum across slots. f32-only path throughout, accumulator order is the
only difference vs the GPU.

## Performance — characterization deferred

The one-shot bench at the Qwen3.6 shape (n_experts=256, top_k=8,
m=512, k=2048, n_tokens=1) measured **10.2 ms / 0.5 GB/s** which is
~20× worse than the indexed-kernel's actual capability. Root cause:
each dispatch creates a fresh `MTLBuffer` for the 150 MiB
all-experts-stacked weight tensor via `newBufferWithBytes_length_options`,
which copies 150 MiB from CPU memory at ~40 GB/s sustained ≈ 4 ms
just for the buffer construction. The kernel itself runs in the
remaining time, but is fully dominated by the alloc.

Real numbers require a persistent-buffer harness:

- pre-allocate `MTLBuffer::newBufferWithBytesNoCopy_*_options` over
  the mmap'd Q4_K weight slab (zero-copy on shared-storage Apple
  Silicon)
- reuse one `MTLCommandQueue` and chain N invocations per command
  buffer (matches what the integrated forward path will do)

That harness is `Stage 4.0` in the optimization plan — same fix that
unblocks the matvec / matmat steady-state numbers for stage 3.1/3.3.

## Selector logic (accepted)

```rust
fn moe_decode_layer(rt: &MetalRuntime,
                    kernels: &MoeKernels,
                    layer: &MoeLayer,
                    input: &[f32; HIDDEN]) -> Vec<f32> {
    // 1. Router → top-k expert IDs + weights, all on CPU since hot loop is small.
    let logits = router_matvec(&layer.router_weights, input);  // [n_experts]
    let mut ids = [0u32; TOP_K];
    let mut wts = [0f32; TOP_K];
    router_softmax_top_k(&logits, TOP_K, &mut ids, &mut wts);

    // 2. Three indexed mat-vecs, gate / up / down. One kernel launch each.
    let gate_out = dispatch_mul_mv_id_q4_k_f32(rt, &kernels.gate, &layer.gate, input,
                                               cast_ids(&ids), N_EXPERTS, INTER, HIDDEN, 1, TOP_K)?;
    let up_out   = dispatch_mul_mv_id_q4_k_f32(rt, &kernels.up,   &layer.up,   input,
                                               cast_ids(&ids), N_EXPERTS, INTER, HIDDEN, 1, TOP_K)?;
    // 3. SwiGLU activation: silu(gate) * up — element-wise on intermediate, [TOP_K, INTER].
    let activated = swiglu(&gate_out, &up_out);
    let down_out = dispatch_mul_mv_id_q4_k_f32(rt, &kernels.down, &layer.down, &activated,
                                               cast_ids(&ids), N_EXPERTS, HIDDEN, INTER, 1, TOP_K)?;
    // 4. Weighted-sum back to residual.
    weighted_sum(&down_out, &wts, TOP_K, HIDDEN)  // [HIDDEN]
}
```
