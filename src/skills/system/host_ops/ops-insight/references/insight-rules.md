# Ops Insight Rules

Keep this skill narrow:

- use `ops_insight` when the goal is a compact decision surface or recurring operational report
- use `reliability_ops`, `incident_response`, `security_posture`, or `recovery_assurance` to gather new domain evidence first
- use `automation_engineering` when a report reveals repeated toil that should become an automation candidate

Conservative interpretation:

- facts come from concrete queue, schedule, host, and shared-kernel evidence
- keep scorecards compact and scoped
- recommendations should point to the right execution skill instead of becoming a vague omnibus report
